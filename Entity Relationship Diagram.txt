A, Identifiers and Relationships
* Step 1: Identify Core Entities
The entities in the Campus Helpdesk System, based on the logical data model, include:
* Users: stores all system users, including administrators, staff members, and request submitters.
* Departments: stores organizational departments responsible for handling support requests.
* Ticket_Categories: stores ticket classifications and routing information for departments.
* Tickets: stores support requests submitted by users.
* Ticket_Comments: stores communications and updates related to support tickets.
* Ticket_Assignments: stores ticket assignment history between staff members.
* Ticket_Status_Logs: stores status change history for auditing and tracking purposes.
* Escalation_Logs: stores escalation records for overdue or unresolved tickets.
* Satisfaction_Surveys: stores user feedback and service quality evaluations after ticket resolution.


* Step 2: Identify Attributes 
* Users: user_id (PK), full_name, email (UNIQUE), password_hash, role, department_id (FK), avatar, is_active, created_at, updated_at
* Departments: id (PK), name, description, manager_id (FK), is_active, created_at, updated_at
* Ticket_Categories: id (PK), name, description, department_id (FK), sla_hours, priority_default, keywords, is_active, created_at, updated_at.
* Tickets: id (PK), ticket_code (UNIQUE), title, description, category_id (FK), submitter_id (FK), assigned_to (FK), priority, status, location, attachment, due_at, resolved_at, closed_at, escalated, created_at, updated_at.
* Ticket_Comments: id (PK), ticket_id (FK), user_id (FK), content, is_internal, created_at, updated_at.
* Ticket_Assignments: id (PK), ticket_id (FK), staff_id (FK), assigned_by (FK), note, assigned_at.
* Ticket_Status_Logs: id (PK), ticket_id (FK), changed_by (FK), old_status, new_status, note, changed_at.
* Escalation_Logs: id (PK), ticket_id (FK), escalated_by (FK), reason, escalated_at.
* Satisfaction_Surveys: id (PK), ticket_id (FK), submitted_by (FK), rating, comment, submitted_at.
* Step 3: Identify Relationships 
User Relationships
* A User can submit multiple Tickets (One-to-Many).
* A User (Staff) can be assigned multiple Tickets (One-to-Many).
* A User can create multiple Ticket Comments (One-to-Many).
* A User can perform multiple Ticket Status Changes (One-to-Many).
* A User can perform multiple Ticket Assignments (One-to-Many).
* A User can submit multiple Satisfaction Surveys (One-to-Many).
* A Department can contain multiple Users (One-to-Many).
* Each Department is managed by one User acting as Department Manager (One-to-One).
Department and Category Relationships
* A Department can manage multiple Ticket Categories (One-to-Many).
* Each Ticket Category belongs to exactly one Department (Many-to-One).
Ticket Relationships
* A Ticket Category can contain multiple Tickets (One-to-Many).
* A Ticket can receive multiple Comments (One-to-Many).
* A Ticket can have multiple Assignment Records (One-to-Many).
* A Ticket can have multiple Status Log Records (One-to-Many).
* A Ticket can have multiple Escalation Records (One-to-Many).
* A Ticket can receive one Satisfaction Survey after resolution (One-to-One).
* Step 4: Constraints and Business Rules
* BR-01: Unique Ticket Code Rule
Every ticket must have a unique ticket_code generated by the system to support identification and tracking.
* BR-02: Department-Based Ticket Routing
Each ticket category must belong to a department. Newly created tickets are automatically routed to the responsible department based on the selected category.
BR-03: Automatic Assignment Rule
When a new ticket is created, the system automatically assigns it to an available staff member with the lowest workload within the corresponding department.
* BR-04: Escalation Rule
If a ticket exceeds its SLA deadline (due_at) and remains unresolved, the system automatically creates an escalation record and notifies the department manager.
* BR-05: Resolution Before Closure Rule
A ticket can only transition to Closed status after it has previously reached Resolved status.
* BR-06: Satisfaction Survey Rule
A satisfaction survey can only be submitted when the corresponding ticket has been resolved or closed.
* BR-07: Internal Comment Visibility Rule
Comments marked as is_internal = 1 are visible only to Staff and Administrators and cannot be viewed by regular users.


B,  Refining the model
 USERS
The USERS table satisfies 3NF because all attributes such as full_name, email, password_hash, role, and department_id depend solely on the primary key user_id. The email attribute is constrained to be unique, preventing duplicate accounts.
DEPARTMENTS
The DEPARTMENTS table stores department-related information. Attributes including name, description, and manager_id are fully dependent on the department identifier id, with no partial or transitive dependencies.
TICKET_CATEGORIES
The TICKET_CATEGORIES table contains ticket classification information. Each category belongs to a specific department and all attributes depend directly on the category identifier.
TICKETS
The TICKETS table is the core transaction entity of the system. All ticket-related information including title, priority, status, due dates, and assignments depends solely on the ticket identifier id. Relationships with users and categories are maintained through foreign keys.
TICKET_COMMENTS
The TICKET_COMMENTS table stores discussions and updates related to support tickets. Each comment is uniquely identified and linked to both a ticket and a user through foreign key relationships.
TICKET_ASSIGNMENTS
The TICKET_ASSIGNMENTS table records assignment history for auditing and workload management purposes. The table avoids redundant storage by separating assignment records from the main ticket entity.
TICKET_STATUS_LOGS
The TICKET_STATUS_LOGS table maintains a historical record of status changes. Each record represents a single transition event and depends entirely on its own primary key.
ESCALATION_LOGS
The ESCALATION_LOGS table captures escalation events triggered when tickets exceed predefined SLA deadlines. This design enables auditability and performance monitoring without duplicating ticket data.
SATISFACTION_SURVEYS
The SATISFACTION_SURVEYS table stores post-resolution feedback provided by users. Survey information is maintained separately from ticket data to preserve normalization and improve reporting flexibility.
